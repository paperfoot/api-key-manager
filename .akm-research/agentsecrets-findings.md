# agentsecrets findings

## What it is

Go cobra CLI (`cmd/agentsecrets/main.go:9`) plus a cloud backend. Commands span account/workspace/project/environment/secret/proxy/agent/log domains (`AGENT.md:184-274`), with 6 auth injection styles (`pkg/proxy/injector.go:16-33`). "Zero-knowledge" here means two things stacked: (1) the **server** holds only AES-256-GCM ciphertext encrypted with a workspace key the server never sees — workspace keys travel between teammates as NaCl SealedBoxes encrypted to each recipient's X25519 public key (`pkg/crypto/crypto.go:166-249`, `docs/ARCHITECTURE.md:52-79`); (2) the **AI agent** only sees key names — values are resolved from the OS keychain inside the proxy/engine just before transport and never returned to the caller (`pkg/proxy/engine.go:218-235`, `pkg/mcp/server.go:150-155` returns `"HTTP %d\n\n%s"` but the credential was injected upstream and any echo is scrubbed). The product is fundamentally a credentialed *forward proxy* with a sync layer bolted on; "zero-knowledge" is mostly a marketing umbrella over the two distinct properties.

## Backing store

OS keychain via `github.com/zalando/go-keyring`, namespaced under service `"AgentSecrets"` with key format `{projectID}:{environment}:{key}` (`pkg/keyring/keyring.go:27`, `:199-204`). Linux/WSL silently falls back to a plaintext-shaped JSON file at `~/.agentsecrets/keyring.json` with mode 0600 — base64-encoded but **not encrypted at rest** beyond filesystem permissions (`pkg/keyring/keyring.go:32-50`, `:143-153`). A separate workspace key plus JWT live in `~/.agentsecrets/config.json` decrypted at login (`docs/ARCHITECTURE.md:236-254`, `pkg/config/config.go:26-44`). Crypto is solid where it's used: X25519 keypair, Argon2id (t=3, m=64MiB, p=4) for password→KEK, AES-256-GCM with 12-byte random nonce prepended to ciphertext (`pkg/crypto/crypto.go:27-35`, `:74-114`). Audit events go to SQLite at `~/.agentsecrets/audit.db` (mode 0700 dir) with a `synced` flag for cloud backfill (`pkg/proxy/audit.go:52-62`, `:81-105`).

## Agent integration

Four surfaces, all in one binary:
1. **MCP server over stdio** (`pkg/mcp/server.go:34-38`, `cmd/agentsecrets/commands/mcp.go:56-63`) — exposes two tools: `list_secrets` (names only, `pkg/mcp/server.go:158-184`) and `api_call` (URL + injections map, `pkg/mcp/server.go:42-71`). `agentsecrets mcp install` auto-writes config to Claude Desktop and Cursor paths (`cmd/agentsecrets/commands/mcp.go:164-192`).
2. **HTTP proxy** on `localhost:8765` listening for `X-AS-Target-URL` + `X-AS-Inject-*` headers (`pkg/proxy/server.go:36-49`, `:95-178`).
3. **OpenClaw exec provider** — JSON in stdin, JSON out (`cmd/agentsecrets/commands/exec.go:15-118`) with a protocol version field; installer drops a 109-line `SKILL.md` skill into `~/.openclaw/skills/agentsecrets/` (`cmd/agentsecrets/commands/mcp.go:207-343`).
4. **Two embedded agent playbooks**: a 692-line operator skill at `integrations/openclaw/SKILL.md` and a 232-line workflow file at `.agent/workflows/agentsecrets.md`. Both are extraordinarily detailed step-by-step runbooks for AI agents, far beyond akm's slim skill.

There is no top-level `agent-info --json` capability manifest — AGENT.md is human-readable Markdown only.

## Things akm doesn't have (potential adoptions)

- **MCP stdio server with `list_secrets`/`api_call` tools** — `pkg/mcp/server.go:34-184`. Letting the agent invoke a tool call instead of shelling out is a real ergonomics win for Claude Desktop/Cursor users. Medium effort (one mcp-rs dep + 200 LOC), high value.
- **Credentialed forward proxy with allowlist + response-body redaction** — `pkg/proxy/engine.go:181-269`, `:24-29`. akm only redacts child stdout for `akm run`; agentsecrets also scrubs upstream API responses that echo the credential back, with `Content-Length` recalculated (`pkg/proxy/engine.go:267`). High effort (proxy server + allowlist storage), high value — closes the API-echo exfil path.
- **Six injection styles (bearer/basic/header/query/body/form) via flags** — `cmd/agentsecrets/commands/call.go:73-106`, `pkg/proxy/injector.go:16-33`. Currently akm only does env-var injection via `run`. Even a `akm call --bearer NAME --url ...` without a full proxy would cover ~80% of use cases. Medium effort, high value.
- **Domain allowlist on outbound calls, checked BEFORE secret resolution** — `pkg/proxy/engine.go:181-199`. The "secret is never even read from keychain if domain isn't allowed" ordering is genuinely good security design. Low effort (TOML list + exact-match check), high value.
- **Storage-mode flag (keychain-only vs .env)** — `pkg/config/config.go:33`, `pkg/secrets/dotenv.go`. akm forces keychain; allowing legacy `.env` export with `akm export NAME > .env` smooths adoption. Low effort, medium value.
- **Environments (dev/staging/prod) as first-class with `secrets diff --from X --to Y`** — `pkg/keyring/keyring.go:199-204` namespaces every key by environment. Diff/copy/merge between envs is genuinely useful and not currently in akm. Medium effort, medium value.
- **SQLite audit store with `synced` flag** instead of JSONL — `pkg/proxy/audit.go:81-129`. Indexed queries (`agent_id`, `domain`, `environment` indices at `:131-139`) make `akm audit list --filter` viable on large logs. Medium effort, medium value. (akm's JSONL is fine for now but caps at ~100k entries before grep gets slow.)
- **Cobra-style `--output-json` flag everywhere** — `cmd/agentsecrets/commands/agent.go:350-360`. akm has `agent-info --json` but command outputs are human-formatted; an agent currently has to parse `akm list` text. Low effort, medium value.
- **Detailed embedded operator playbook** — `integrations/openclaw/SKILL.md` is 692 lines vs akm's roughly 100-line skill. Specifically the "Commands the agent CAN run / CANNOT run / NEEDS USER APPROVAL FOR" matrix (`integrations/openclaw/SKILL.md:608-657`) is well thought through and matches the "no prompts" design philosophy. Low effort, medium value to crib that tri-state matrix.
- **`mcp install` auto-writes to known IDE config paths** — `cmd/agentsecrets/commands/mcp.go:164-204`. Even without an MCP server, `akm skill install --target claude-desktop` writing to `~/Library/Application Support/Claude/claude_desktop_config.json` is a paper cut worth removing. Low effort, low value (today; high if/when akm adds MCP).

## Things akm does better

- **Macro/macOS-only scope keeps the security story tight.** agentsecrets falls back to a base64-shaped JSON file on Linux/WSL (`pkg/keyring/keyring.go:32-50`, `:143-153`) which is decidedly NOT keychain-grade despite the README promising "Linux Secret Service" (`README.md:308`).
- **No silent network egress.** agentsecrets has a telemetry sync that pushes command counts to an internal API every 24h on CLI exit (`cmd/agentsecrets/commands/root.go:68`, `pkg/telemetry/telemetry.go`); akm is offline-first.
- **One binary, zero accounts, no cloud login required.** agentsecrets demands `agentsecrets login` for almost every operation that touches secrets (`cmd/agentsecrets/commands/root.go:92-100`); akm just works on first run.
- **Semantic exit codes (0/1/3/6) and typed errors.** agentsecrets' `main.go:9-12` collapses every error into `os.Exit(1)` — agents can't differentiate "key not found" from "auth failed" from "network down".
- **`agent-info --json` capability manifest.** agentsecrets has no machine-readable manifest at the top level; agents must scrape AGENT.md/SKILL.md.
- **Pre-commit guard for staged `sk-`/`ghp_` blobs.** No equivalent in agentsecrets.
- **No prompts, ever.** agentsecrets uses `fmt.Scanln` for y/N confirmations in `agent token revoke`, `agent delete`, etc. (`cmd/agentsecrets/commands/agent.go:243-247`, `:272-278`, `:309-314`) — these block unattended agent execution and are gated only by a `--confirm` flag.

## Footguns / bugs / weak spots

- **README and AGENT.md lie about session-token auth on the proxy.** README claims "Session token: generated at proxy startup, required on every request" (`README.md:291`) and AGENT.md repeats it (`AGENT.md:85`). But `pkg/proxy/server.go:95-178` (the actual `handleProxy`) never checks any session token — any process on localhost can hit `127.0.0.1:8765/proxy` and exfiltrate via an allowlisted domain. This is the headline security feature and it's not implemented.
- **keychain-auth is disabled in v1.3.0.** The README and ARCHITECTURE.md describe process-level binary-hash verification as a core property (`docs/ARCHITECTURE.md:311-356`, `README.md` security table), but the middleware short-circuits with `return nil` and the real logic is commented out (`cmd/agentsecrets/commands/root.go:132-180`, `cmd/agentsecrets/commands/env.go:161-191`, `cmd/agentsecrets/commands/exec.go:120-140`). Any process running as the user can call go-keyring directly.
- **No SSRF protection in the proxy.** README claims private IPs / non-HTTPS targets are blocked (`README.md:289`); the engine only does case-insensitive exact-host allowlist comparison (`pkg/proxy/engine.go:187-198`) and the forwarder doesn't even pre-validate the URL scheme (`pkg/proxy/forwarder.go:24-46`). If `evil.com` is allowlisted, `http://evil.com` works, and the request follows redirects via the default `http.Client` (`pkg/proxy/engine.go:97-99`) — meaning a 302 to `http://10.0.0.1/internal` would be followed by Go's default redirect policy. No `CheckRedirect` is set.
- **Allowlist is exact-match only.** `pkg/proxy/engine.go:188-192` does `strings.ToLower(raw) == targetDomain`. ARCHITECTURE.md (`:141`) brags "subdomains must match (e.g., api.stripe.com does not automatically allow evil.api.stripe.com)" — true, but the inverse is also true: you can't allowlist `*.stripe.com`. Operationally painful.
- **Linux/WSL keyring file is base64, not encrypted.** `pkg/keyring/keyring.go:143-153` writes JSON with base64 values at 0600. Anyone reading that file (root, backup tools, accidental git add) gets plaintext secrets.
- **Workspace key cached in plaintext in `~/.agentsecrets/config.json`.** `docs/ARCHITECTURE.md:253-254` admits this: "decrypted at login using your private key... cached here for performance." File mode 0600 (`pkg/config/config.go:116`) is the only protection. So one filesystem read = all team secrets decryptable.
- **MCP `parseInjections` accepts arbitrary user-controlled body paths.** `pkg/mcp/server.go:196-228` happily passes any `body:foo.bar.baz` path to the injector, which then mutates JSON anywhere in the request body (`pkg/proxy/injector.go:79-132`). Combined with no schema validation in `api_call`, a prompt-injected agent can wrap `body:auth.token` around credentials and ship them to an allowlisted-but-attacker-friendly endpoint.
- **Telemetry leaks active environment name to The Seventeen's servers** by default with no opt-out flag I could find in this read pass (`docs/ARCHITECTURE.md:360-375`, `pkg/telemetry/telemetry.go`).
- **`agentsecrets sync` returns mock data.** `pkg/proxy/engine.go:303-312`: `if len(e.RevokedIDs) == 0 { e.RevokedIDs = []string{"rev_test_5k2m88"} }` — revocation list sync is a stub.
- **Allowlist stored in OS keychain rather than config.** `pkg/keyring/keyring.go:286-309`. Means a `secrets pull` workflow on a new machine has to also bootstrap allowlist, and the agent can't read it without a keychain-auth prompt (when keychain-auth is re-enabled).

## Verdict

Adopt three things, ignore the rest. (1) **A `akm call` subcommand with the six injection styles** (`pkg/proxy/injector.go`, `cmd/agentsecrets/commands/call.go`) — this is the highest-leverage adoption; most "agent makes API request" use cases collapse to one CLI invocation that never returns the value. (2) **Response-body redaction in `akm run`'s child stdout/stderr scanner, extended to upstream HTTP responses if/when akm gains a `call` subcommand** — `pkg/proxy/engine.go:243-269` is 25 lines of code that close a real exfil path. (3) **The tri-state "agent can run / needs approval / cannot run" matrix from `integrations/openclaw/SKILL.md:608-657`** ported into akm's skill — it's the right mental model for a no-prompt agent CLI and akm's current skill doesn't articulate it. Everything else (workspaces, environments, cloud sync, MCP, agent identity tokens) is product surface area you should explicitly *not* pursue — it dilutes the single-binary keychain-CLI ethos that makes akm faster and more trustworthy than this much larger system.
